# Audit Log

Dated reports from periodic content audits. One file per week, generated by the `/drift-audit` skill.

## Files

- `drift-YYYY-Www.md` — weekly drill drift audit. Sample size 10 (default), rotating-cursor selection so all 131 drills get audited within ~13 weeks.
- `.cursor.json` — internal state for the rotating cursor. Schema: `{ "audited": { "<drill-slug>": "<YYYY-MM-DD>" } }`. Tracked in git so audit history persists across machines.

## How a report gets produced

`/drift-audit` reads `.cursor.json`, picks the 10 oldest-audited (or never-audited) drills, runs `/drill-review` on each, writes the punch-list per drill plus a recurring-patterns section to a new dated report file.

## How to read a report

1. **Summary** — quick triage. Clean / minor / critical counts.
2. **Recurring patterns** — checks that failed on ≥3 drills this week. These are the actionable items: a single drill failure is noise; a pattern across multiple drills is signal.
3. **Per-drill findings** — punch list per drill. Use to fix specific drills.

## Acting on a report

Recurring patterns drive corpus-wide fixes (e.g., the May 2026 finding that drill JSONs are accent-stripped → corpus restoration pass). Per-drill findings drive targeted edits when the drill comes up in normal use.

Reports are append-only. Don't edit historical reports — they're a record of what the corpus looked like at a point in time.
